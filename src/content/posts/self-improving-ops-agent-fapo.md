반복되는 사내 운영 업무를 대신 처리하고, **그 경험으로 스스로 절차를 개선하는 자기개선 루프(FAPO)**를 운영 에이전트 '조이(zoee.alter)'에 적용해봤다. 목표는 단순 자동화가 아니라 한 단계 위 — *일한 기록을 모아 자기 절차의 빈틈을 찾아 스스로 개선안을 내는* 운영 에이전트다.

## 왜 — 두 종류의 비효율

운영 업무에는 두 가지 비효율이 있었다.

1. **매일·매주 똑같이 반복되는 일.** 운영 인프라 건강 확인, 배포·임베딩 문의 응대, MCP 에러를 모아 이슈로 올리기, 하루 업무 정리 — 내용은 매번 비슷한데 사람이 매번 손으로 한다.
2. **한 번 해결한 문제가 휘발된다.** 어떤 사고를 해결해도 그 절차가 머릿속에만 남고 글로 안 남아서, 비슷한 일이 또 오면 처음부터 다시 한다. 즉 **'학습'이 누적되지 않는다.**

그래서 목표를 한 단계 위로 잡았다.

> 단순히 일을 대신하는 봇이 아니라, 일한 기록을 모아 자기 절차의 빈틈을 찾아 스스로 개선안을 내는 운영 에이전트.

## FAPO — 스스로 나아지는 루프

FAPO(*Fully Automated Prompt Optimization*)는 Cisco가 공개한 자동 프롬프트 최적화 개념(Apache 2.0)이다. 핵심은 **"AI가 일하다 실패하면, 그 실패를 단계별로 기록해 → 어디서 틀렸는지 짚고 → 그 부분만 고친다"**. 우리는 이걸 '프롬프트 튜닝'이 아니라 **운영 절차(워크플로) 개선**에 가져왔다.

조이는 하나의 에이전트지만 **두 개의 저장소로 이루어진 한 몸**이다. 사람으로 치면 몸과 뇌다.

| 구성 | 역할 | 비유 |
|---|---|---|
| **hermes** | 메시지를 받고(Slack·Teams·CLI) 도구를 쓰고 정해진 시각에 일을 실행. 실제로 일하는 쪽 | 몸 (실행) |
| **zoee-knowledge** | 무엇을 어떤 절차로 해야 하는지, 시스템이 어떻게 생겼는지를 정의. 판단 기준이 담긴 쪽 | 뇌 (지식) |

몸이 일한 흔적을 로그로 남기면 → 별도의 '검토자(옵티마이저)'가 그 로그를 읽어 개선안을 제안하고 → 사람이 승인하면 뇌(절차)가 업데이트된다.

![FAPO 개선안 처리 프로세스 — 실제 업무 수행(사용자) → Slack·Teams·CLI 로그 기록 → 로그 검토 → 개선안 PR → 사람(조이) 승인 → zoee-knowledge 절차에 반영. 반영된 지식이 다시 업무 수행을 지원하는 순환.](https://svc-fnf-dcs-ai-s3.s3.ap-northeast-2.amazonaws.com/aie-app-agent/blog/joy-fapo/joy-loop-1c09a76c.png)

### 5단계 루프

1. **수집** — 새 로그 중 실패·부분성공(`fail`/`partial`)만 추린다. (로그는 매일 자동 생성되며, 사내 정보라 git에 안 올리고 로컬에만 둔다)
2. **진단** — 실패를 *어느 단계에서 어긋났나*로 묶는다 → 검색(정보를 못 찾음) / 추론(찾았는데 판단이 틀림) / 형식(결과 형태가 틀림) / 연쇄(앞 실패가 뒤로 번짐).
3. **제안** — 실패 단계가 *고칠 대상*을 결정한다. 검색 실패 → 지식 구조·링크 보완, 추론·연쇄 → 절차서·스킬 보완, 형식 → 문구만. 원칙은 *가장 작은 정당한 수정*. 과거에 거절된 시도는 먼저 조회해 반복하지 않는다(anti-repeat).
4. **검증** — 수정한 절차로 다시 돌려본다(replay). 평가 케이스와 비교해 **점수가 오르고 기존이 안 망가질 때만** 채택.
5. **PR** — 변경안을 올리고, 사람이 승인하면 절차가 갱신된다.

> **콜드스타트와 N=5 규칙** — 평가 케이스가 적으면(워크플로당 5개 미만) replay 채점을 못 믿는다. 그래서 5개 미만이면 **cold 모드**(채점 없이 명백한 수정 + 평가 케이스 보강 제안만), 5개 이상이면 **warm 모드**(재채점)로 동작. 검증 안 된 변경이 함부로 들어가는 걸 막는 장치다.

### 역할 분리 — 누가 무엇을 쓸 수 있나

자기개선이 폭주하지 않도록 권한을 셋으로 갈랐다. **일하는 쪽이 자기 절차를 마음대로 못 고치게** 하는 게 핵심이다.

| 주체 | 쓰기 권한 |
|---|---|
| **조이 (일하는 쪽)** | **로그만** — 자기 절차 수정 금지 |
| **옵티마이저 (검토자)** | **개선안 PR 제안만** — 직접 반영 불가 |
| **사람** | **머지(최종 승인) — 유일한 게이트** |

## 이번 실험 — 관측에서 제안까지

핵심 질문: **"매 사고를 사람이 직접 고치기(인라인)" vs "검토자가 로그를 모아 개선(배치/옵티마이저)"** 중 무엇이 나은가. 옵티마이저를 실제로 한 번 돌려봤다.

### 어떻게 검증했나

6/24~6/26 사흘간 실제 작업 로그 8건이 쌓였다. 이걸로 옵티마이저를 **섀도우 실행**(제안만 하고 실제 반영은 안 하는 모드)했다. 평가 케이스가 아직 0개라 점수를 매기는 대신(cold 모드) "어디서 반복 실패가 나나"를 진단하게 했고, 그 산출물을 같은 기간 사람이 손으로 한 인라인 수정과 나란히 비교했다.

### 옵티마이저가 실제로 잡아낸 것

8건의 로그에서 사람이 매번은 못 보는 빈틈·패턴들이 나왔다.

**(가) "잘 돌던 인프라 검진"에 숨은 빈틈 — silent success.** 인프라 아침 검진은 *원래부터 매일 하던 작업*이다. 그런데 로그를 보니 점검만 한 게 아니라 실제로는 **쌓인 MCP 세션 259개를 정리하는 조치까지 했는데 그게 절차서엔 안 적혀 있었다.** 성공했고 아무도 불평하지 않아(silent success) 사람은 못 알아챈 빈틈.
→ **제안:** "캐파 경고 + 세션 과다 보유자 발견 시 정리" 단계를 절차에 명문화 — 다음에 누가 해도 재현되게.

**(나) 흩어진 장애 대응이 사실 한 워크플로 — unmapped 반복.** 서로 다른 두 건의 메모리 부족(OOM) 대응이 매번 "분류 안 됨"으로 기록됐는데, 둘 다 같은 근본원인 이슈로 수렴하고 절차도 거의 같았다. **한 건이면 일회성 버그, 합치면 하나의 워크플로.**
→ **제안:** `incident-response`를 새 워크플로로 승격(제보 → 메트릭 실측 → 세션 교차확인 → 근본원인 분류 → 즉효 조치 → 이슈 발행) + "복원 경로에서 Redis TTL 건드리지 말 것" 되돌림 금지 메모.

**(다) 봇 자신의 복구도 반복 패턴.** 게이트웨이 장애 2건(스레드 맥락 끊김 / recap 파이프라인 장애)이 표면은 다른데 같은 복구 절차를 밟았다.
→ **제안:** `gateway-recovery` 새 워크플로 (봇 자신의 자산이라 별도 레포로).

**(라) 기존 절차의 도구 제약.** 배포 문의 처리에서 응답 안 한 문의를 *이모지 리액션*으로 가리려다 실패(우리가 쓰는 도구가 리액션을 못 읽음).
→ **제안:** "리액션이 아니라 스레드 답글 유무로 판별"을 절차에 박기. *도구의 제약을 절차서에 명시해두면 다음에 같은 함정에 안 빠진다.*

### 결론 — 인라인 vs 배치는 상보적

| 방식 | 강점 |
|---|---|
| **인라인 (사람이 즉시)** | 맥락을 다 알아 깊고 빠름 — 긴급·심층에 강함 |
| **배치 (검토자가 주기적)** | 여러 사건에 걸친 패턴을 잡고 사람 부담이 적음 — **사람이 놓치는 사각(silent success·반복 패턴)을 잡음** |

**둘은 경쟁이 아니라 역할 분담.** 위 (가)~(다)는 *한 건만 봐선 안 보이고, 불평이 없어서 사람이 못 챙기던* 빈틈인데, 배치 검토자가 로그를 모아 보고서야 드러났다.

### 검토자가 자기 자신을 고친 사례

배치 옵티마이저가 어떤 구체적 수정(위 '라')을 '개선안'이 아니라 '평가 케이스'로 잘못 분류해 묻어버린 적이 있다. 사람이 같은 로그를 수동 검토한 결과와 대조하다 이를 발견했고, **옵티마이저의 지침 자체를 고쳤다**("구체적 수정은 평가 케이스로 묻지 말고 개선안으로 올려라"). *검토자를 검토하는 2차 층(메타리뷰)*이 실제로 작동한 첫 사례다.

## 사람의 결정이 어떻게 '데이터'가 되나

개선안은 **GitHub PR**로 올라온다. 여기서부터가 핵심이다.

```text
   옵티마이저 개선안
        │
   GitHub PR (open)
        │     ← 사람은 Slack에서만 결정한다
        ├─ "머지"          → merged          (채택)
        ├─ "수정 후 머지"  → merged+revision (보완 채택)
        └─ "닫기"          → closed          (거부)
        │
   Notion DB (상태 자동 미러)
        │
   데이터셋 (채택·거부·교정 분포) → 메타리뷰 (옵티마이저 개선)
```

1. **상태관리는 GitHub가 한다.** PR은 세 상태로 끝난다 — 머지(채택) / 닫기(거부) / 수정 후 머지(보완 채택). 이 상태 전이가 곧 사람의 판단 라벨이다.
2. **사람은 Slack에서만 결정한다.** GitHub를 직접 만지지 않는다. Slack에서 "PR #1 머지해 / 이 단계 빼고 머지 / 닫아"라고 하면 실행체가 GitHub에서 그대로 실행한다. 창구는 Slack 하나뿐.
3. **PR 상태는 Notion DB로 자동 미러된다.** GitHub Action이 PR 열림·닫힘마다 Notion '옵티마이저 제안' DB에 상태를 갱신한다. *(이번 주 첫 PR로 이 자동 동기화를 실제로 검증했다.)*
4. **쌓인 상태가 데이터셋이다.** "이 종류 제안은 늘 채택/거절된다"는 분포가 *옵티마이저가 얼마나 쓸 만한지*를 말해준다. 특히 **'수정 후 머지'의 원안↔최종본 차이**가 가장 값진 신호 — "방향은 맞는데 늘 이걸 고쳐야 한다"는 교정 패턴. 메타리뷰가 이 데이터로 옵티마이저를 다듬는다.

> 한 줄로: **GitHub = 상태관리 백엔드 / Slack = 사람의 단일 결정 창구 / Notion = 결정이 쌓이는 데이터셋 대시보드.** 사람이 누른 merge/close 한 번이 그대로 학습 데이터가 된다.

## 이 루프를 떠받치는 것들

루프가 돌려면 ① 매일 일하고 로그를 남기는 실행체와 ② 절차가 쌓이는 지식기반이 필요하다.

**hermes — 매일 일하는 실행체.** 정해진 시각에 도는 cron 작업들이 올라가 있다. 대표적으로 **daily-recap**은 하루 작업(여러 PC 세션 + git)을 업무일지로 정리하는 3단 파이프라인 — ① 수집(판단 없이 사실만) → ② 정제(맥미니 로컬 모델 MLX로 1차 요약 — API 비용 0, 사내 데이터 외부 유출 없음) → ③ 종합(고성능 모델이 교차검증 후 기록 + *자기개선용 로그를 emit*). **'싼 일은 로컬, 판단은 고성능 모델'**로 단계를 나눈 게 설계 포인트다. 이번 주에 ③에 "FAPO 로그 자동 생성"을 얹어, 루프의 입력을 만들었다.

**zoee-knowledge — 그래프 지식기반.** "그냥 문서 폴더"가 아니라 절차·규칙을 마크다운으로 쓰되 파일끼리 **위키링크(`[[파일명]]`)로 연결**해 전체가 하나의 그래프를 이룬다. 사람이 읽고 git으로 버전 관리하기 쉽고, 깨진 링크는 자기개선 루프가 검출해 보강한다.

## 배운 점

1. **자동화의 진짜 가치는 '반복 제거'가 아니라 '학습 누적'에 있다.** 일을 대신하는 봇은 시간을 아끼지만, 거기서 멈추면 교훈이 매번 휘발된다. 로그→개선→절차반영 루프를 붙여야 한 번의 해결이 영구 자산이 된다.
2. **권한을 가르는 게 자율성의 전제다.** 일하는 쪽은 로그만, 검토자는 제안만, 최종 승인은 사람 — 이 분리가 있어야 '스스로 개선'을 안심하고 켤 수 있다.
3. **비용은 단계별로 나눠 쓴다.** 분량 많은 1차 요약은 로컬 모델(MLX, 비용 0), 판단이 필요한 마지막만 고성능 모델.
4. **사람과 자동화는 경쟁이 아니라 상보.** 인라인(즉시·심층)과 배치(패턴·사각 탐지)는 잘하는 영역이 다르다. silent success처럼 사람이 구조적으로 못 보는 빈틈은 자동 검토만 잡는다.
