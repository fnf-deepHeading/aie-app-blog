이번 주 집중 주제는 **AI 업무 환경 만들기**였다. Claude Code·Codex·MCP·개인 에이전트 Hermes(이하 알터)를 *쓰고는* 있었지만, 업무 흐름 자체는 여전히 수작업이었다 — 업무 읽기, 맥락 정리, 프롬프트 투입, 결과 확인을 매번 직접. 목표는 개별 업무를 하나씩 자동화하는 게 아니라, **업무 자동화를 만들고 운영하는 상위 계층**을 세우는 것이었다.

## 관점 전환 — 한 단계 더 추상화된 코딩

- 기존 코딩 = 서비스를 직접 만드는 코딩
- 앞으로의 코딩 = AI 엔진이 서비스 개발 업무를 *대신 수행*하도록 만드는, 한 단계 더 추상화된 코딩

즉 AI가 일하고 검증하고 보고하는 흐름 자체를 코드와 명세로 작성한다. 최종 목표는, 모든 업무를 직접 처리하는 사람이 아니라 **알터가 준비한 실행안과 승인 큐를 보고 판단만 하는 사람**이 되는 것.

처음 그린 흐름은 이랬다:

```text
업무 발생
 → 업무 유형 분류
 → 필요한 컨텍스트 수집
 → Claude Code 또는 Codex에 작업 위임
 → MCP·Hermes로 도구 연결
 → 스크립트 기반 검증
 → 위험 작업은 승인 큐로 분리
 → 결과를 PR·issue·문서·리포트로 남김
```

## 왜 그동안 자동화가 잘 안 됐나

- **개별 작업 중심.** 전체 흐름을 하나의 루프로 안 보고 중간 작업만 AI에 위임 → 자동화가 단발성 호출에 머묾.
- **수동 컨텍스트 주입.** 관련 issue·PR diff·실패 로그·DB 스키마·테스트 명령·프로젝트 규칙을 매번 직접 찾아 붙임.
- **특정 상황 과적합.** 입력·맥락이 조금만 바뀌어도 새 프롬프트를 다시 작성. 재사용 가능한 워크플로 수준으로 추상화하지 못함.
- **낮은 신뢰도 → 잦은 개입.** 자동화 구조가 부실 → 불신 → 재확인의 악순환. 심하면 자동화가 *오히려 추가 업무*로 전락.

> 문제의 본질은 AI 도구 자체의 부족이 아니라, 도구를 안전하게 운용할 **명세·검증·가드레일·승인 구조의 부재**였다.

## 일괄 구축의 실패와 '설정 병목'의 발견

방향은 지금도 옳다고 본다. 문제는 실행이었다. 거버넌스·자동화·운영·역할 에이전트·워크로그 파이프라인까지 **한 번에** 구축하려다, 1인 동시 작업의 결과로 폴더 구조가 비대해졌다 — 활성 파일 80개 초과. 만들고 보니 *자동화 운영 계층을 운영하는 일 자체가 새로운 부담*이었다.

여기서 멈추고 물었다. **"내가 진짜 불편했던 게 뭐였나?"**

답은 자동화 흐름이 아니라 **설정**이었다. 에이전트 자체가 흩어지고(Codex·Claude Code·Hermes), 한 에이전트 안에서도 글로벌·프로젝트별·로컬 설정으로 또 분산된다. 관리도, 초기 세팅도 어렵고, 한번 적용하면 굳어 유연한 변경이 곤란하다. 자동화 운영 계층은 그 위에 얹을 *지붕*인데, 정작 *바닥(설정)*이 흔들리고 있었다.

## 문제 재정의 — 거울이 아니라 지도

자동화 흐름을 짓기 전에, 분산된 설정을 모으고 관리하는 **표면**부터 만들기로 했다. 야심 찬 운영 레이어는 archive로 보류, 활성 파일 80여 개 → 20개로 축소. 대신 단순한 관측소 하나를 세웠다.

핵심은 매니페스트 파일 하나, `config-map.json`. 성격은 **위치 지도** — "어떤 도구의 / 어떤 스코프(enterprise·user·project·local)의 / 어떤 설정이 / 어느 경로에 있는지"만 기록한다. 수집 스크립트가 이 지도를 읽어 현황(존재 여부·최종 수정 시각·크기)을 한 화면에 렌더한다.

여기서 가장 중요한 원칙이 나왔다 — **map-not-mirror, 거울이 아니라 지도.**

- 진실은 각 도구의 홈에 있는 실제 파일.
- 관측소는 위치를 가리키고 현황만 보고할 뿐, **사본을 들고 동기화하지 않는다.**
- 사본을 SSOT로 삼는 순간 생기는 문제: 비밀 누출, 원본과의 불일치, 한번 굳으면 안 바뀜. 원칙의 핵심은 *그 문제를 애초에 만들지 않는 것*.

추가 원칙은 **시크릿은 메타데이터만.** 관측소는 토큰·키 같은 값을 절대 읽지도·보여주지도·저장하지도 않는다. "존재 / 어느 스코프 / 어느 경로 / 회전 필요" 같은 메타만 다루고, 화이트리스트에 적은 안전한 필드만 노출, 나머지는 마스킹. 토큰이 박힐 수 있는 필드(env·headers·url·args)는 화이트리스트에서 제외. 기본값은 닫힘(fail-closed).

## "그냥 한 폴더에 모으면 되잖아"가 틀린 이유

관측소를 만들자 당연한 반론이 떠올랐다. 흩어진 게 문제라면 한 폴더에 잘 모으면 되지, 굳이 지도를 따로 그릴 필요가 있나?

진지하게 검토한 답은 **"한 폴더로 모으는 건 애초에 불가능"**이었다.

- 각 도구가 자기 홈을 하드코딩해서 읽는다(Claude Code `~/.claude`, Codex `~/.codex`, Hermes 세 군데). 한 폴더에 모아둬도 도구들은 거기를 안 본다.
- enterprise 정책 파일은 *소유권이 없다.* 회사가 배포하는 관리 설정은 옮길 수 있는 대상이 아니다.
- 시크릿 파일은 repo로 복사하는 것 자체가 금지 — 모으는 행위가 곧 사고.
- 스코프 레이어링(enterprise > user > project > local)은 버그가 아니라 *기능*이다. 한 폴더로 합치면 "어느 스코프가 이긴다"는 우선순위 정보가 오히려 깨진다.

> 이 흩어짐은 정리를 못해서가 아니라, 도구와 조직이 강제하는 **줄일 수 없는(irreducible) 흩어짐**이다. 따라서 모으는(mirror) 게 아니라 가리키는(map) 것이 정답이다.

또 하나 — **"한 번 잘 정리하면 되잖아"도 오류다.** 잘 정리하는 건 1회 스냅샷일 뿐, 진짜 문제는 **드리프트**다. 도구 업데이트로 설정 경로가 늘고, MCP가 새로 깔리고, enterprise 정책이 바뀌며 구성은 계속 움직인다. 관측소는 같은 지도를 다시 읽어 시간축으로 비교해 *드리프트를 가시화*한다. 한 번 정리로는 대체 불가다.

## 데스크탑 앱으로 — project-mint

읽기 측이 검증되자, 실제로 들여다보고 다룰 표면이 필요했다. 가벼운 데스크탑 앱으로 확장했다(이름 `project-mint`, 스택 Tauri·React). 하나의 매니페스트 위에 두 표면을 올린 구조다.

- `configs/` — `config-map.json` 매니페스트. 위치의 단일 출처. 비밀도, 머신 경로도 없음.
- `app/` — Tauri·React 데스크탑 GUI. 내부에 Rust 읽기 엔진.
- `core/` — TypeScript 엔진. 같은 읽기를 한 번 더 구현. MCP 서버도 여기서 노출.

**읽기 엔진을 Rust(앱용)와 TypeScript(headless·MCP용)로 두 번 구현한 건 의도적 선택**이다. 보통은 중복이라 피하지만, 두 엔진이 같은 매니페스트를 *계약*으로 공유하니 한쪽을 바꾸면 다른 쪽과 어긋나는지가 드러난다. (다만 공유 fixture 검증은 미구현이라, 한쪽만 고치지 않게 조심하는 건 현재 사람 몫이다.)

앱이 지키는 불변식 네 가지:

1. **map-not-mirror** — 진실은 각 도구의 실제 파일. 앱은 가리키고 보고만.
2. **시크릿은 메타데이터만** — 존재/스코프/경로/회전 플래그만. 값은 읽지도·보여주지도·저장하지도 않음.
3. **순수 관측소** — 앱은 읽기 전용, 내부에 LLM 없음. 실제 설정 변경은 외부 에이전트(Claude Code·Codex)에 위임, 앱은 MCP로 맥락만 전달.
4. **데몬도, 자동 시작도, 무인증 서버도 없음.**

마지막 불변식이 중요했던 건 반면교사 때문이다. 참고한 오픈소스 중 UI 패턴은 좋으나 보안 모델이 정반대인 사례가 있었다 — 무인증으로 모든 인터페이스에 서버 개방, 임의 파일 읽기, 시크릿 노출. **UI 메커닉은 차용하되 보안 모델은 통째로 버렸다.**

초기엔 앱이 직접 설정을 고치는 적용/제거 기능을 넣어 구현·테스트까지 했지만, 도구마다 어댑터를 다 만들려니 조합 폭발 + 시크릿 안전 처리가 반복적으로 헛돌았다. 결국 그 기능은 archive하고, **변경은 에이전트가 판단·수행, 앱은 수집·조회와 에이전트 핸드오프만** 담당하도록 피벗했다. 핸드오프는 MCP 서버로 구현 — 도구 6개(`get_inventory`·`get_skills`·`search`·`read_doc`·`get_running_agents`·`get_usage_stats`), 전부 **메타데이터와 경로만** 반환하고 파일 내용·시크릿은 한 글자도 안 넘긴다.

## 이 앱이 굳이 필요한가

앱을 만들고 가장 날카로운 질문을 스스로에게 던졌다. 정직한 결론은 이렇다.

> **읽기 전용 지도 렌더링에만 머물면 이 앱은 과설계다.** 그건 인벤토리 파일 하나를 예쁘게 복제한 것에 지나지 않는다. 마크다운이나 MCP가 못 하는 일을 실제로 얹을 때만 앱은 정당화된다.

지금 사용 패턴이 그 조건을 충족하는지 따져봤다.

- **메모리 관리**가 대표적이다. 메모리는 정적으로 읽는 게 아니라 항목끼리 `[[링크]]`로 잇고, 추가·편집·검색하는 *인터랙션*이 핵심 — 정적 마크다운이 구조적으로 못 하는 영역이다.
- **눈으로 반복해서 들여다보는** 일이다. 4축(메모리·워크플로·구성·문서) 칸반에서 접고 펼치고 드릴다운하며 드리프트를 한눈에 보는 건 텍스트 덤프보다 GUI가 분명히 낫다.
- **드리프트 추적**은 1회 정리로 대체되지 않는다.

그래서 선을 분명히 했다 — 앱이 다시 '읽기 전용 지도 렌더링'으로 되돌아가면 과설계다. 마크다운이 못 하는 일을 실제로 얹어야 "굳이 필요하다"가 성립한다. 이 기준을 통과하지 못하면 앱을 접는 게 맞다.

## 더 깊은 원인 — 프롬프트·스킬을 자산으로 다루지 않았다

한 주를 돌아보니 "왜 자동화가 안 됐나"보다 한 겹 더 들어간 원인이 있었다. **그동안 프롬프트와 스킬을 자산으로 여기지 않았다.** 매번 같은 프롬프트를 다시 입력, MCP를 다시 연결, 스킬을 다시 태그해 워크플로를 즉석에서 만들고 — 그 대화 세션(업무 흐름)을 한 번도 분석하지 않았다. 프롬프트를 깎고 마크다운을 다듬어 다음번에 재사용하는 일을 안 했다.

그래서 세운 가설: **업무에서 어떤 도구를 어떤 맥락으로 어떤 흐름에 썼는지 가시화 → 그걸 토대로 프롬프트·스킬을 깎는 루프를 돌리면, 결과 신뢰도와 자동화 루프 정확도가 재귀적으로 상승할 것.** `get_usage_stats`로 세션에서 어떤 스킬·MCP가 실제로 얼마나 쓰였는지 집계하게 만든 것도 이 가설 검증의 첫걸음이다.

이번 매니페스트 검증에서 이 방식의 효과를 실제로 확인했다. 세 도구의 설정 위치를 정확히 적기 위해 deep-research 워크플로를 돌렸고(공식 문서로 독립적 ground truth 생성 → 매니페스트와 1:1 대조 → 적대적 재검증), 오류 여럿을 잡았다 — 예: Codex의 `~/.codex/skills`는 공식 스캔 경로가 아니라 래퍼가 만든 비공식 경로였고(공식은 `~/.agents/skills`), "Hermes가 런타임에 프로젝트 컨텍스트를 로드하지 않는다"는 기존 메모가 사실이 아니었다. *즉흥적으로 한 번 묻고 끝냈다면 못 잡았을 것들.*

## 이번 주에 실제로 도달한 곳

처음 그렸던 "자동화 운영 계층"에는 도달하지 못했다. 대신 그 토대가 되는 관측소와, 그걸 다룰 앱·에이전트 핸드오프 통로를 만들었다.

- 세 도구 × 스코프에 흩어진 설정의 위치를 매핑하는 매니페스트(`config-map.json`)와 검증된 도구별 스펙.
- 같은 읽기를 Rust(앱)·TypeScript(headless)로 구현한 이중 엔진. 매니페스트가 둘을 묶는 계약.
- 4축 칸반·드릴다운을 갖춘 읽기 전용 데스크탑 앱(`project-mint`).
- 에이전트에게 같은 지도를 넘기는 MCP 서버(도구 6개, 전부 메타데이터·경로만).
- 비싸게 배운 불변식 네 가지(map-not-mirror, 시크릿 메타only, 순수 관측소, 무데몬·무인증).

솔직히 처음 계획과 비교하면 절반은 접고 절반은 다른 길로 왔다. 하지만 진짜 수확은 산출물이 아니라 두 깨달음이다. 하나는 **토대(흩어진 설정을 보는 표면)가 없으면 그 위 자동화는 모래 위에 짓는 집**이라는 것. 다른 하나는 **한꺼번에 다 짓지 말고, 손으로 두 번 반복해서 아플 때만 구조를 더하라**는 것. 실제로 "2회 아플 때만 구조 추가"를 불변식으로 박아 뒀다.

## 알터 고도화 — 워크플로 4개

토대 작업과 별개로, 알터에 업무 워크플로 4개를 추가·개선했다.

### 1. 아침 업무 브리핑

### 4. 데일리 AI 메모리 요약

여러 도구(Claude Code·Codex·Hermes)에 흩어진 auto-memory를 매니페스트의 위치 정보로 스캔해, 그날 새로 쌓인 메모리만 한국어로 요약. 위 관측소의 첫 실사용 사례다.

![데일리 메모리 다이제스트 — 여러 도구·프로젝트에 흩어진 새 메모리를 모아 출처별로 한국어로 요약한 봇 메시지](https://svc-fnf-dcs-ai-s3.s3.ap-northeast-2.amazonaws.com/aie-app-agent/blog/hoy-alter/hoy-4-memory-3e0ed7a2.png)

이 워크플로의 흐름은 이렇다 — map-not-mirror 원칙이 그대로 실행에 옮겨진 모습이다:

```text
Hermes cron (매일 18:00)
   │
   ▼
cron 에이전트 (LLM)
   │ ① get_recent_memories 호출
   ▼
workflow-manager MCP ──② 위치 조회──▶ 매니페스트 SSOT (config-map.json)
   │ ③ 스캔
   ├─▶ Claude Code  ~/.claude/projects/*/memory (전 프로젝트)
   ├─▶ Codex        ~/.codex/memories
   └─▶ Hermes       (로컬 메모리 경로)
        └ 제외: AGENTS.md·CLAUDE.md·SOUL.md·인덱스 등 (entity=memory만)
   │ ④ 노트 메타 목록 → 에이전트
   │ ⑤ 본문 필요시 read_doc (.md만)
   ▼
⑥ 한국어 요약 → ⑦ Slack 전송
```

남은 건 둘이다. 만든 MCP 서버를 Claude Code·Codex에 실제로 등록해 대화로 검증하는 일(headless 스모크 테스트는 통과), 그리고 위 가설(세션을 분석해 프롬프트·스킬을 깎는 재귀 루프)을 실제 업무에 돌려보는 일. 다음 주의 출발점이다.
