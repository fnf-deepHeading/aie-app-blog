증상은 단순했다. 멀쩡히 돌던 서버가 며칠에 한 번 OOM으로 죽고, 재시작하면 또 며칠 잘 돌았다. 부하가 튀는 시간대와도 상관없었다. 천천히, 꾸준히 메모리가 차올랐다.

## 천천히 차오르는 그래프

메모리 사용량 그래프가 톱니가 아니라 **계단**이었다. 요청이 끝나도 내려오지 않고, 한 단 한 단 올라가다 한계에서 떨어졌다(=재시작).

전형적인 누수 패턴이다. 어딘가에서 객체가 GC 대상이 되지 못하고 붙잡혀 있다는 뜻.

```bash
# 힙 스냅샷을 시간차로 떠서 무엇이 안 사라지는지 본다
node --inspect …
# Chrome DevTools → Memory → 두 스냅샷 비교(Comparison)
```

## 범인: 끝나지 않는 세션

스냅샷을 비교하니 MCP 세션 객체가 계속 쌓이고 있었다. 세션을 열 때 등록한 핸들러·타이머·연결이 세션이 끝나도 해제되지 않았다. 클라이언트가 깔끔하게 종료하지 않은 세션은 영원히 살아 있었다.

핵심은 두 가지였다:

1. **생성한 자원은 생성한 곳에서 정리한다.** 세션 종료 경로뿐 아니라, 비정상 종료·타임아웃 경로에서도 같은 정리를 보장.
2. **세션에 수명을 준다.** 일정 시간 활동이 없으면 강제로 거둬들이는 타임아웃을 둬서, 클라이언트가 인사 없이 사라져도 서버가 잊지 않게.

> 누수는 대개 "만드는 코드"가 아니라 "치우는 코드가 없는 것"이다.

## 남은 것

계단이 톱니로 바뀌었다. 재시작 주기가 사라졌고, 같은 노드에서 훨씬 오래 안정적으로 돌고 있다.

교훈은 코드가 아니라 습관이었다. 이제 세션·연결·타이머처럼 수명이 있는 자원을 열 때, 닫는 경로를 같은 PR에서 함께 적는다 — 정상 종료만이 아니라 모든 종료에서.
