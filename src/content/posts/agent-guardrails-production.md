데모는 누구나 만든다. 어려운 건 그 에이전트를 실제 사업부 업무 흐름에 꽂는 일이다. 사내 에이전트를 운영에 올리며 세운 가드레일을 공유한다.

## 1. 출력은 항상 구조화한다

모델의 자유 텍스트를 그대로 다음 단계로 흘려보내지 않는다. 도구(tool) 스키마를 강제하고, 검증을 통과한 것만 실행·렌더한다.

```ts
const result = await client.messages.create({
  model: 'claude-opus-4-8',
  tools: [{ name: 'propose_action', input_schema: schema }],
  tool_choice: { type: 'tool', name: 'propose_action' },
  messages,
});
// 스키마 검증 실패 시 실행하지 않고 사람 확인 단계로
```

덕분에 "갑자기 형식이 깨진 응답이 그대로 흘러가는" 류의 사고가 사라졌다.

## 2. 실행 전에 사람이 끼어들 자리를 만든다

에이전트가 무언가를 **제안**하는 것과 **실행**하는 것 사이에 게이트를 뒀다. 영향이 큰 작업(데이터 변경·외부 발송)은 사람이 승인해야 통과한다.

- 읽기·요약·조회는 자동.
- 쓰기·발송·삭제는 제안 → 승인 → 실행.
- 모든 실행은 누가·언제·무엇을 감사 로그에 남긴다.

> 자동화의 신뢰는 "다 맡길 수 있다"가 아니라 "위험한 건 멈춰 선다"에서 나온다.

## 3. 비용과 품질을 한 화면에서 본다

호출 한 건의 토큰·지연·도구 사용·실패 여부를 모두 로깅해 대시보드로 묶었다. 캐싱이 안 먹는 프롬프트 하나가 비용의 큰 몫을 먹고 있던 걸 여기서 잡았다.

## 남은 것

품질은 만족스럽지만, 평가가 여전히 수동이다. 모델이나 프롬프트를 바꿀 때마다 사람이 표본을 읽는다. 자동 평가 파이프라인이 다음 숙제다.
