운영 에이전트 '키티(Kitty.Alter)'에서 진행한 한 주(6/22~26) 정리. 가장 큰 흐름은 **업무 일지가 주간 회의록·상반기 리뷰 같은 후속 자료의 기반이 될 수 있게 일관성을 잡는 일**이었다. 그 외에도 도메인 가이드 체계, 2개 소울 구조, 운영 업무 자동화 시도 3건, 응답 속도 개선이 함께 진행됐다.

## 도메인 가이드 체계 정착

키티가 현업 도메인 질문에 충분히 답하지 못하던 사고가 누적돼서, 봇이 참고할 도메인 자료를 본업 프로젝트 안에 통일된 자리로 박았다.

- 7개 본업 프로젝트에 `docs/agent-guide.md` — PRD와 별도로 봇 답변용 도메인 정리(룰·로직·FAQ·외부 노출 범위).
- 워크스페이스 junction 도입 — 기존 격리 룰 폐지, 8개 본업 폴더를 봇이 모두 read·write 가능하게.
- 정기 전달 파일 슬랙 첨부 룰 — 말만 하고 첨부가 안 되던 사고를 3단계 업로드 패턴으로 해결.

## 2개 소울 구조 도입

발화의 결을 보고 봇이 자동으로 모드를 전환하게 했다.

- 기본 키티(default·라우터) + 경리 키티(예산·정산 모드) 두 모드 정의.
- 경리 모드의 진실원천을 `work-budget` 스킬로 통합 — 특화 룰 4개를 SOUL에서 제거하고 한 곳에서만 관리.

## 운영 업무 자동화 시도 — 3건, 그리고 그 한계

현업 담당자가 주기적으로 하는 운영 업무 3건을 키티에게 위임 시도했다. 셋 다 일부 단계는 자동화됐지만 **완전 자동화까지 도달한 건은 없다.** 그리고 각 한계의 이유가 서로 다르다는 게 핵심이다.

### ① 마일리지 정기전달 — 외부 인프라 제약

자료 전달은 자동화했지만 자료 *생성*은 자동화하지 못했다.

- 자료가 있으면 슬랙 첨부 자동 전달, 없으면 담당자 PC 세션에 생성 요청 → 처리 후 봇이 첨부 전달.
- 못 한 이유: 생성 파이프라인이 Snowflake 브라우저 SSO(사람 로그인)와 Windows Excel 매크로(특정 PC 한정)에 의존. 리눅스 봇이 원격 무인 생성 불가.

### ② 유사상품 월간 모니터링 — 품질 검증에 사람 개입 필수

매월 첫째 목요일 자동 trigger되는 cron으로 박았다. Step 1·2는 자동, 사이에 사람이 한 번 개입한 뒤 Step 3~6 자동.

- 못 한 이유: Step 2에서 다운로드한 이미지 중 비정상 이미지(뒷면·착용샷·디테일컷·멀티뷰 등)를 사람이 직접 골라 거르는 검증이 필수. **모델 학습 품질에 직결되는 단계라 자동 판별에 맡겼다가 결과를 망칠 위험.**

### ③ 카테고리 키워드 업데이트 — 운영 안전성 신중함

리마인드 cron만 박고 작업 자체는 담당자가 직접 한다.

- 안 한 이유: 키워드 업데이트는 실 운영 DB에 직접 쓰기가 일어난다. 잘못된 키워드가 운영에 박힐 위험이 있어 **담당자 본인이 검수하고 본인 손으로 박는 게 안전.**

> 세 건의 한계 패턴이 다르다 — **외부 인프라 제약 / 품질 검증 사람 개입 / 운영 안전성 신중함.** 자동화의 한계는 기술 문제만이 아니다.

## ★ 업무 일지 일관성 — 맥락을 위한 룰 정착

이번 주의 핵심. 현업 담당자가 직접 짚은 사고에서 시작했다.

> 업무 로그를 그날그날 업무만 보고 쌓다 보니 맥락도 끊기고 쓰는 단어도 달라서, 일관성이 없어 이걸 기반으로 주간자료를 AI가 만들기 힘들어한다.

일지 row가 *그날 데이터*(Task List·git commit·todo·스터디 노트)만 보고 작성돼서, 같은 프로젝트인데도 매번 다른 결로 쌓였다. 용어·표현이 흩어졌고 "왜 이 작업을 했나"가 빠진 row가 많았다. 결과적으로 주간자료 생성 LLM이 일관성을 못 잡고 헷갈렸다.

### 말머리를 다는 기준 — `[성격 · 단계]`

일지 row마다 붙는 `[성격 · 단계]` 말머리는 임의 라벨이 아니다. 출발점은 단순한 질문이었다 — *"그 일을 하는 사람은 실제로 어떤 흐름으로 일하는가?"* 머신러닝 엔지니어라면 문제정의에서 시작해 데이터를 모으고, 모델을 만들고, 평가하고, 배포·운영으로 흐른다. 그 직무의 실제 작업 흐름을 단계로 두고, 각 작업이 그 중 어디인지로 말머리를 단다.

판단은 세 단계다.

1. **프로젝트 성격 파악** — ML / SW개발 / 운영 / 데이터분석 / 리서치·학습. 폴더명이나 표면 단어로 단정하지 않고 PRD·코드 같은 진실원으로 확인한다.
2. **그 성격의 일반 업무 단계를 떠올린다.**
   - **ML**: 문제정의 → 데이터수집·EDA → 데이터준비 → 모델링·실험 → 평가·검증 → 배포·서빙 → 모니터링·운영
   - **SW 개발**: 요구사항·기획 → 설계 → 구현 → 테스트 → 배포 → 유지보수
   - **운영**: 이슈감지·모니터링 → 룰·정책개선 → 버그수정·장애대응 → 자동화·구성변경 → 배포·적용
   - **데이터 분석**: 문제정의 → 자료조사·검토 → 분석·실험 → 정리·문서화·공유
   - **리서치·학습**: 자료조사·검토 → 학습·이해 → 정리·공유
3. **각 작업을 단계에 매핑** → 말머리 = `[성격 · 단계]`.

원칙은 둘이다. 한 row에 여러 단계를 섞지 않는다 — 한 작업이 두 단계에 걸치면 분리한다. 그리고 이슈·블로커는 "누가 처리하나"가 아니라 "그 이슈가 어느 작업 내용에 관한 거냐"로 해당 단계에 붙인다. 작업을 평면적으로 나열하면 단계가 섞여 맥락이 뭉개지기 때문에, **"이 작업이 그 직무의 어느 단계인가"를 먼저 판단해 단계가 안 섞이게 구조화하는 것**이 말머리의 목적이다. 아래 룰들(특히 룰 2의 `[성격 · 단계]` 두 부분 강제)은 모두 이 체계 위에서 작동한다.

### ① 최근 row 컨텍스트 조회 단계 신설

일지를 작성하기 직전, 그날 기록될 프로젝트별로 일지 DB에서 **같은 프로젝트의 최근 14일 row를 조회해 LLM 입력에 같이 넣는** 단계를 추가했다. 가져오는 건 같은 프로젝트 row의 제목·업무내용·단계 말머리. 새 row를 쓸 때 그 흐름을 보고 용어·서술 결을 이어가게 한다. (5분 캐시로 반복 호출 비용은 작다.)

### ② 일관성 룰 7개 — 4회 검증·정정 사이클로 정착

스킬에 룰만 박는 것으로는 한계가 명확했다. 본 일지 6/14~20을 테스트 DB에 **4번 다시 쌓으며**, 담당자가 검증하고 사고를 짚을 때마다 룰을 보강했다. 사이클 4회 끝에 룰 7개가 정착했다.

| 룰 | 사고 | 정정 |
|---|---|---|
| **1. 단계 말머리는 기존 패턴 우선** | 운영 row가 일지엔 `[운영 · 룰·정책개선]`인데 봇이 스킬 예시 따라 다르게 박음 | 스킬 예시 vs 실제 패턴 충돌 시 일지 우선. 예시는 가이드라인일 뿐 |
| **2. `[성격 · 단계]` 두 부분 필수** | 말머리가 `[리서치]`로만 박혀 단계 누락 | 두 부분 모두 강제(`[리서치 · 학습·이해]`) |
| **3. 본문 첫 문장에 '왜' 필수** | 7개 row가 무엇만 적고 왜가 빠짐 | "다운타임 우려로 배포 방식을 무중단으로 전환"처럼 동기부터. 단서는 피드백 메모·commit 본문·todo |
| **4. 봇 내부 의사결정·메타 정보 본문 금지** | row 본문 끝에 봇의 분류·제외 판단이 노출됨 | 봇이 자기 판단은 본문에 안 씀. 사람이 일지 볼 때 "왜 이게 적혀있지" 헷갈림 |
| **5. 봇 인지용 메모리 본문 금지** | 봇 헷갈림 방지용 메모리 텍스트가 매 row 본문에 박힘 | 메모리는 분류·매핑 시 참조만. 본문엔 작업 의미·결과만 |
| **6. WHY vs WHERE TO USE 구분** | commit의 "~를 위해" 문구를 봇이 진짜 WHY로 해석 | "~위해 / ~가 쓸 수 있도록"은 활용처·부산물. 진짜 WHY는 사람의 요청·문제. 단서 부족하면 가짜 WHY 안 씀 |
| **7. 본문 간결성** | commit log의 모든 기술 디테일(style ID·파라미터·파일명)이 본문에 박힘 | 3~5문장 자연어, 의미 단위만. 구체 ID·파라미터·라이브러리명 제외 |

정정 사이클은 v1(기본 룰) → v4(룰 7까지)로 진화했다. 매 버전마다 새 사고가 드러났고, 그게 다음 룰이 됐다.

### 이 작업의 의미

- **스킬에 룰을 박는 것만으로는 한계가 있다.** 실제 row를 사람이 검증하고 짚어야 룰이 정확해진다 — 예측이 아니라 *사고 사례 학습*이다.
- **입력 소스 자체가 사고의 근원일 수 있다.** commit 메시지의 추론(룰 6)이 일지로 흘러 의도와 어긋났다. 그래서 글로벌 git 규칙(모든 프로젝트의 commit은 WHAT·HOW·결과만, WHY 추론 금지)까지 거슬러 올라가야 근본 해결이 됐다.
- **담당자의 짚음이 룰 정착의 핵심 입력.** 룰 1→7은 봇이나 메인 혼자서는 못 만들었을 것들이다.

## 응답 속도 개선

단순 일정 발화에도 답이 느리다는 지적에서 시작한 다층 개선.

- prompt caching TTL 5분 → 1시간(간헐적 트래픽 캐시 적중률 향상).
- Notion 헬퍼에 5분 파일 캐시 도입 — 실측 **638ms → 32ms(20배)**. 매번 GET으로 data source를 resolve하던 걸 하드코딩으로 우회.
- 결과: 단순 발화는 두 번째 호출부터 즉답.

흥미로운 건, 이 응답 속도 인프라(Notion 캐시)가 일관성 룰의 '최근 row 컨텍스트 조회'와 단순 발화 답 속도를 **둘 다** 받쳐준다는 점이다.

## 이번 주 핵심 배움

- 운영 업무 자동화의 한계는 기술 문제만이 아니다 — 외부 인프라 제약·품질 검증 사람 개입·운영 안전성 신중함, 셋 다 다른 결의 제약.
- 일관성 작업은 단순한 룰 박기가 아니라 *실제 결과 검증 → 사고 짚기 → 룰 보강* 사이클이다. 사람의 짚음이 정착의 핵심 입력.
- 문제의 근본은 스킬 룰을 넘어 입력 소스 자체(commit 메시지)에 있을 수 있다. 글로벌 규칙까지 거슬러 올라가야 한다.
- SOUL·스킬 룰만으로 LLM이 안 따르면, 시스템 측(코드로 강제 차단)까지 가야 영구 해결된다.
