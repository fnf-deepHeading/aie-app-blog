처음엔 우리가 다 만들어 줬다. 사업부에서 "이런 화면 하나 필요해요" 하면, 받아서 만들고, 올리고, 링크를 줬다. 요청이 열 건일 땐 굴러갔다. 백 건이 되자 우리가 병목이 됐다.

## 문제: 사람이 게이트인 구조

AI로 무언가를 만드는 비용은 빠르게 0에 수렴하는데, 그걸 **배포하고 공유하는 비용**은 그대로였다. 모든 배포가 우리 손을 거쳐야 했기 때문이다.

- 정적 결과물 하나 공유하려고 인프라 담당이 붙는다.
- 사업부는 GitHub 계정도, 배포 권한도 없다.
- 누가 무엇을 배포했는지 추적이 안 된다.

> 플랫폼의 목표는 우리가 더 많이 만드는 게 아니라, 우리 없이도 굴러가게 하는 것이었다.

## 결정: 산출물의 종류로 경로를 나눈다

모든 배포를 한 방식으로 처리하려다 매번 막혔다. 그래서 산출물 형태로 경로를 갈랐다.

1. **정적 결과물**(HTML/빌드 산출물)은 파일 업로드만으로 공유 URL이 나온다. 코드도, 계정도 필요 없다.
2. **동적 앱**(서버가 붙는 것)은 저장소 기반으로 배포한다.

대부분의 사업부 요청은 1번이었다. 가장 흔한 경우를 가장 가볍게 만들자 요청의 절반 이상이 우리 손을 떠나 스스로 굴러가기 시작했다.

### 셀프서비스에는 가드레일이 따라온다

직접 배포하게 하려면, 아무나 아무거나 보지 못하게 막아야 했다. 그래서 처음부터 같이 붙인 것들:

- 임직원 명부 기반 접근 제한 — 특정 사용자·팀·본부만 보게.
- 개인/시스템 API Key 분리와 감사 로그.
- 해외 법인(HQ/CN) 모드와 이메일 코드 로그인.

기능보다 이 권한·감사 레이어가 더 오래 걸렸다. 하지만 이게 없으면 셀프서비스는 켤 수 없었다.

## 남은 것

전환 뒤 한 달여 동안 배포는 수백 건으로 늘었고, 배포한 사람도 100명을 넘겼다. 더 중요한 건 그 대부분이 우리를 거치지 않았다는 점이다.

대신 새 병목이 생겼다. 배포 경로가 여러 갈래로 난립하기 시작했고, 데이터 저장소(S3·DB) 세팅은 여전히 어렵다. 다음은 이 배포 로직을 한 구조로 합치고, 저장소 발급까지 셀프서비스로 넘기는 일이다.
