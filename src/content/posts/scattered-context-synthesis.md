매주 우리는 파트 주간보고를 만든다. 파트 밖(본부장·이사님·타 팀)도 읽으므로 "사람이 직접 쓴 듯한" 품질이 필요하다. 이걸 AI로 자동화하며, 우리는 품질의 레버가 더 큰 모델이나 더 정교한 프롬프트가 아니라 **구조**에 있다는 걸 거듭 확인했다.

구조는 두 층위에 있었다. 하나는 **입력의 구조**(일지를 어떻게 쓰나) — 같은 에이전트·같은 프롬프트로 돌려도 일지를 의미 단위로 잘 끊어 적었는지에 따라 결과가 역전된다. 이건 일지 작성 관점에서 따로 다룰 예정이다. 이 글은 다른 한 층위, **파이프라인의 구조** — 여러 소스에 흩어지고 사람과 프로젝트가 다대다로 얽힌 맥락을 *어떻게 한 곳에서 엮나* — 를 우리 실측과 함께 기록한다.

## 문제: 맥락은 흩어져 있고, 주체는 다대다다

두 가지 구조적 어려움이 있다.

**첫째, 한 업무의 맥락이 한 곳에 없다.** 한 사람의 한 작업도 일지(무엇을 했나의 서술), git(실제 커밋·수치), 슬랙(의사결정·블로커·대기), 노션 업무DB(상태·스프린트)에 *조각으로 흩어져* 있다. 어느 하나도 완전하지 않다.

**둘째, 사람과 프로젝트가 다대다다.** 우리 파트는 5명이 14개 활성 프로젝트에 걸쳐 일한다. 한 프로젝트를 여러 명이, 한 사람이 여러 프로젝트를. 예를 들어 핵심 플랫폼 프로젝트 하나는 세 명이 공동으로(개발 둘·인프라 하나) 굴리고, "개인 AI 봇 개발"은 다섯 명이 *각자 자기 봇*을 만든다. 한 사람의 한 주를 보면 메인 프로젝트 하나에 운영성 잡무 서넛이 얹힌다. 결국 "누가 무슨 프로젝트에 얼마나 기여했나"를 한 보고로 엮어야 하는데, 어느 한 축(사람만, 또는 프로젝트만)으로 줄 세우면 다른 축이 깨진다.

> 산출 품질은 입력의 구조·품질에 좌우된다 — 입력이 부실하면 산출도 그렇다. 그런데 그 입력은 여러 소스에 흩어져 있고 소스마다 품질이 제각각이다. 그렇다면 핵심 질문은 — **어떻게 흩어진 입력들을 한 맥락으로 엮고, 부실한 소스는 서로 보완시키나?**

## 소스마다 묶이는 기준이 다르다 - 어떤 소스는 사람별로, 어떤 소스는 프로젝트별로

| 소스 | 담는 것 | 기준 |
|---|---|---|
| 개인 일지(노션) | 무엇을 했나·이슈/블로커 | 사람 (조직 공유) |
| git | 실제 커밋·정량·구현 | 사람(author) · 물리적으로 각자 환경 |
| 슬랙 | 의사결정·대기·블로커 | 채널(참여자) |
| 업무DB(노션) | 프로젝트 상태·스프린트 | 프로젝트 |
| 프로젝트 목록(정본) | 활성 프로젝트·담당·정체 | 파트 전체 정본 |

이렇듯 소스마다 기준이 다르다. 일지·git·슬랙은 *사람/채널* 단위로, 업무DB·프로젝트 목록은 *프로젝트* 단위로 존재한다. 게다가 git은 *물리적으로* 각자 PC·클론에 묶여 있다. **이렇게 기준이 갈리는 것이 이후 모든 설계 결정의 뿌리다.**

## 한 프롬프트로는 안 됐다 — 왜 워크플로인가

처음엔 *한 개의 큰 프롬프트*로 다 했다. 한 번의 호출이 수집·판단·작성·검토를 전부 맡는 방식 — 빠르지만 천장이 낮았다.

**프롬프트 하나로 돌릴 때**
- 5명 × 여러 프로젝트를 한 맥락에 욱여넣으니 *남의 일·깊은 디테일*이 먼저 뭉개진다 (맥락도 주의도 한정).
- 자기가 쓴 걸 자기가 검토 — 무른 자가검토라 빠진 깊이·지어낸 것을 못 잡는다.
- 다대다인데 *순차*다 — 하나 끝나야 다음.
- 작성 규칙이 쌓일수록(우리는 16개) 한 프롬프트가 그걸 다 지키지 못한다.

**역할별 에이전트(워크플로)로 바꾼 뒤**
- 각 에이전트가 *한 가지*에만 집중 → **깊이**(격리)
- *독립* 비평 에이전트가 담당자처럼 따짐 → **품질**
- 독립 단위(사람·프로젝트)는 *동시에* → **속도**
- 작성·비평·수정이 같은 규칙(FRAMING)을 공유 → **일관**

전환 자체에도 함정이 있었다. "워크플로로 *재조립*한다"면서 그 16개 작성 규칙을 두어 줄로 요약해 넣었더니 품질이 곧장 무너졌다 — 규칙을 *원문 그대로* 되살리고서야 돌아왔다. **누적된 작성 규칙은 코드만큼이나 핵심 자산이다 — 재사용해야지 재작성하면 안 된다.**

## 에이전트를 어떻게 엮었나

그래서 이걸 한 프롬프트가 아니라 **역할이 다른 AI 에이전트들의 조립**으로 만들었다. 전체 모양은 이렇다 — 핵심은 사람별로 한 번(나눔), 프로젝트별로 한 번(합침), **두 번 모양을 바꾼다**는 것.

에이전트 구성에서 결정한 게 셋이다.

1. **병렬로 펼치기** — 사람 단위는 서로 독립이라, 한 명을 끝내고 다음으로 가지 않고 5명을 *동시에* 합성한다. 그래서 사람이 늘어도 벽시계가 거의 안 는다.
2. **프로젝트별 파이프라인** — 합치는 단계도 프로젝트마다 독립이라 줄 세우지 않는다. 한 프로젝트가 '작성→비평→수정'을 도는 동안 다른 프로젝트도 같이 흐른다.
3. **비평을 *별도 에이전트*로** — 작성한 에이전트가 자기 글을 자기가 검토하면 무르다. 그래서 *원본 소스와 그 프로젝트 담당자 입장을 시뮬레이션하는 독립 비평 에이전트*가 "이 작업이 실제로 어떻게 끝났나, 빠진 깊이는 없나, 지어낸 건 없나"를 따로 따진다. **이 독립 비평이 품질의 핵심**이었다(뒤에서 다룬다).

아래 네 결정은 이 구조 위에서 *데이터를 어느 축으로 잇느냐*의 선택이다.

## 네 개의 설계 결정 포인트

흩어진 다대다 맥락을 엮기 위해 네 가지 포인트를 세워보았다.

### 1. 어디까지 나누고 언제 합치나

소스가 사람 단위와 프로젝트 단위로 갈리니, 자연스러운 2단계가 나온다: **사람 단위로 나눠 합성 → 프로젝트 단위로 병합.** 사람별 작업은 병렬로 돌릴 수 있다.

그런데 함정을 만났다 — **이중 압축.** 처음엔 사람별 단계에서 *이미 공유용으로 압축*해 넘겼다. 그러자 병합 단계가 받을 때 디테일이 이미 증발해 있었고, **없는 깊이는 복구할 수 없었다.** 단일 단계로 원본을 곧장 변환하던 기존 도구보다 오히려 나빠졌다.

한 멤버의 추천 모델 작업으로 직접 비교해봤다. **같은 입력·같은 에이전트, 압축 시점만 바꿨다.** 사람별 단계에서 미리 공유용으로 압축한 경우(전)와, 깊이를 그대로 들고 가 합치는 단계에서만 추상화한 경우(후)의 *실제 산출*이다.

**전 — 사람별 단계에서 미리 압축 (이중 압축)**

```text
추천 모델 — 완료
· 추천 로직을 보강하고 협력사 매칭을 개선
· 추천 API를 타 팀에 인계
```

**후 — 압축을 합치는 단계로 미룸 (깊이 보존)**

```text
추천 모델 — 완료
· 디자인이 비슷한 상품을 더 잘 찾도록 추천 점수 보강
    - 디자인 정보가 빈 후보가 점수에서 부당하게 밀리던 문제
      → 그 후보는 빈 항목을 빼고 채점(중립 처리)
    - 시연 기준 가중치 = 이미지 유사도 0.8 / 디자인 라벨 0.2
· 협력사명을 한글·영문 양쪽으로 매칭
    - 사내 시스템의 영문명을 연결 → 'YS KOREA' = '와이에스코리아'
    - (git에서 보강) 원가 데이터 5,192행, 영문명 98.9% 채움
· 추천 API를 타 팀이 직접 운영하도록 인계 (사용 가이드·배포 절차)
```

'전'은 한 주 작업을 두 줄로 뭉갠다 — 읽는 사람은 *무엇을* 했는지 알 수 없다. '후'는 같은 작업에서 의사결정(왜 그 가중치인지, 어떤 문제를 어떻게 풀었는지)까지 남는다. 그리고 핵심은 — 일단 '전'처럼 뭉개고 나면, 합치는 단계가 아무리 똑똑해도 '후'를 되만들지 못했다는 것이다.

> 교훈: 나누되 *압축하지 마라.* 나누는 단계는 깊이를 보존만 하고, 추상화는 합치는 단계에서 *한 번만* 한다. **나눔은 일찍, 압축은 늦게.**

### 2. 분리인가 중앙인가

git처럼 물리적으로 흩어진 소스를, 각자 PC에서 *분산* 수집해 모을 수도 있었다. 하지만 그건 5대 PC에 코드 배포·인증정보·전처리 공통화라는 운영 부담을 낳는다.

핵심 관찰: **한 사람(파트 PM)의 접근 권한이 이미 대부분을 포괄한다** — 일지(조직 공유 전원), git(author별로 한 곳에서), 슬랙(대부분 채널). 그래서 **중앙 수집**을 택했다. 못 보는 일부(내가 안 들어간 슬랙 채널)는 그 멤버의 일지가 커버한다. 타인의 git은 후속으로, 공용 폴더에 *1회 클론 후 갱신만* 하는 식으로 중앙에 끌어오면 된다.

그래도 되나? 측정해봤다. *일지 밖 소스(git·슬랙)에서만 새로 건진 항목 수*를 세니 — 일지를 깊게 쓰는 사람은 22건 중 단 2건, 얕게 쓰는 사람은 7건 중 4건이었다. 즉 **git·슬랙 보강은 적응적이다**: 일지가 충실하면 거의 0, 부실하면 메운다. 분산 수집의 부담을 5대에 지우지 않아도, 중앙 한 곳에서 본 것만으로 대부분이 채워졌다. (이건 앞의 전제와 이어진다 — 입력이 좋을수록 다른 소스로 보완할 게 줄어든다.)

> 교훈: 분리/중앙은 취향이 아니라 **접근 권한의 위상이 정한다.** 한 노드가 충분히 포괄하면 중앙이 압도적으로 싸다(배포 0·인증정보 1벌).

### 3. 단계 사이에 무엇을 주고받나

분해와 병합을 잇는 것은 **중간 산출물의 스키마**다. 우리는 사람별 단계가 떨구는 *개인 위클리 문서*(JSON)의 형식을 정해뒀다: `사람 → 프로젝트별 → 항목[제목·상태·요점(계층 중첩 허용)·출처]` + 실무 체크리스트 + 생성 과정 메모. 병합 단계는 원본 일지를 다시 읽지 않고 이 문서만 읽고 합친다.

실제로 떨어지는 한 토막은 이렇게 생겼다:

```jsonc
{
  "person": "○○", "projects": [{
    "project": "유사상품 추천 모델",          // ← 합칠 때 쓸 프로젝트 키
    "items": [{
      "title": "추천 로직 보강",
      "status": "완료",
      "points": [                            // 계층 중첩 허용 — 깊이 보존
        { "t": "협력사 한·영 양방향 매칭",
          "sub": ["영문명 컬럼 조인", "5,192행·98.9% 채움"] }
      ],
      "sources": ["일지", "git"]             // ← 어디서 왔는지(보완·검증용)
    }],
    "checklist": ["⚠️ 기준 이미지 PLM/견적서 우선순위 확인 필요"],
    "trail": ["일지 6/15·6/19 항목을 한 작업으로 병합"]
  }]
}
```

`sources`가 "일지+git"인 게 보일 것이다 — 같은 항목을 두 소스가 *교차 보강*한 흔적이다. `checklist`/`trail`은 합칠 때 그대로 위로 올라간다.

이게 핵심인 이유: 나누는 기준(사람)과 합치는 기준(프로젝트)이 다른데, 그 둘을 잇는 정해둔 공통 형식이 바로 이 문서다. 게다가 파일로 떨어지니 재사용·전파·디버깅이 덤으로 따라온다.

이 `checklist`·`trail`은 최종 보고서에서도 살아남아, 산출이 *이사님 결정 / 실무 체크리스트(담당자 후속) / 생성 트레일(부정합·편집 판단)* 세 갈래로 분리돼 나온다 — AI가 무엇을 어떻게 합쳤는지 사람이 추적·신뢰할 수 있게.

> 교훈: 다대다 파이프라인의 진짜 설계물은 단계가 아니라 단계와 단계 사이에 정해둔 형식이다.

### 4. 누락을 어떻게 막나

초기에 한 핵심 프로젝트가 보고에서 **통째로 사라지는** 사고가 났다. 사라진 건 한 멤버의 메인인 *추천 모델 프로젝트*였다. 병합 단계가 *개인 위클리에 등장한 프로젝트만* 모았기 때문이다 — 그 주 기여 조각이 이름이 비슷한 *별개의 '운영' 프로젝트*로 잘못 흘러들어가자, 정작 모델 프로젝트는 어느 섹션에도 안 남았다. 시스템은 *"그런 활성 프로젝트가 애초에 존재한다"는 사실 자체를* 몰랐다. 누가 무엇을 올렸느냐에만 기대니, 아무도 안 올리거나 엉뚱한 데 올리면 그 자리는 조용히 빈다.

해결은 공식 프로젝트 목록을 먼저 뼈대로 세우는 것이었다. 파트의 활성 프로젝트들을 기준 목록으로 깔아두고, 각 프로젝트가 보고서에 반드시 들어갈 자리를 갖게 한 뒤, 사람별로 올라온 기여를 그 자리에 끼워 넣는다. 덤으로, 이름이 비슷한 다른 프로젝트(운영 vs 모델링)가 엉뚱하게 합쳐지던 것도 — 목록이 둘을 다른 프로젝트로 구분해주니 — 막혔다.

> 교훈: 다대다에서 *올라온 것만* 모으면 조용한 누락이 생긴다. **무엇이 존재해야 하는가를 먼저 고정**하고 기여를 채워라.

## 구현 — 코드로는 이렇게

전체 조립은 세 가지 기본 동작의 조합이다: `agent()`(에이전트 하나 실행, 결과를 *정해둔 형식*으로 받음), `parallel()`(여럿 동시), `pipeline()`(항목마다 단계별 흐름).

**Stage 1 — 사람별 합성을 한 번에 병렬로:**

```js
const 개인위클리들 = await parallel(
  로스터.map(사람 => () =>
    agent( stage1프롬프트(사람),
           { schema: 개인위클리_형식, model: 'sonnet' } )   // 추출·합성은 싼 모델
  )
)
```

**Stage 2 — 프로젝트별 파이프라인 (작성 → 비평 → 수정):**

```js
const 프로젝트_spec = await pipeline( 활성프로젝트들,

  // ① 작성
  proj => agent( 작성프롬프트(proj),
                 { schema: 섹션_형식, model: 'opus' } ),

  // ② 독립 비평(담당자 시뮬) → 지적 있으면 ③ 수정
  async (초안) => {
    const 판정 = await agent( 비평프롬프트(초안), { schema: 판정_형식 } )
    return 판정.통과 ? 초안
                     : agent( 수정프롬프트(초안, 판정), { schema: 섹션_형식 } )
  }
)
```

읽을 포인트 넷:
- `parallel`은 **동시에** 펼치고(사람 수만큼), `pipeline`은 항목마다 **독립적으로 단계를 흐른다** — 한 프로젝트가 ②비평 중이어도 다른 프로젝트는 ①작성 중일 수 있다(줄 안 섬).
- `schema`로 결과 형식을 강제한다 — 모델이 형식을 어기면 자동 재시도. 그래서 앞서 말한 *단계 사이 계약*(개인위클리 형식)이 깨지지 않는다.
- `model`을 단계마다 지정한다 — 추출은 Sonnet, 작성·비평은 Opus.
- 작성·비평·수정 에이전트는 모두 같은 **작성 규칙(FRAMING)**을 주입받는다(충실도·계층·수치 보존 등). 이게 품질을 만든다 — 그리고 처음엔 이걸 빠뜨려서 한 번 무너뜨렸다.

## 한 장 만드는 데 얼마가 드나

실제로 돌린 두 주의 수치다(매번 5명 × 등록 프로젝트 14개 확인). 한 장에 **약 20분**.

| 측정 | 가벼운 주 | 무거운 주 |
|---|---|---|
| 묶인 항목 | 56 | 92 |
| AI 작업 수 | 26 | 30 |
| 토큰 | 126만 | 157만 |
| 추정 비용 | 약 1.5만 원 | 약 1.9만 원 |

- **묶인 항목** = 한 주 일을 중복·잡무 정리해 묶은 단위. 한 사람당 주 열 건 안팎.
- **AI 작업 수** = 사람별 합성 5 + 프로젝트별 (작성→비평→수정) + 정리. 사람 수가 아니라 **프로젝트 수 × 검토 3라운드**에 비례한다(사람은 병렬이라 둔감).
- **모델** = 사실 추출·합성하는 앞단은 빠르고 싼 **Sonnet**, 판단이 필요한 작성·비평은 **Opus**. 비싼 모델을 *어디에 쓰느냐*가 곧 비용 설계다.

## 이 경험을 어디에 쓰나

우리가 푼 건 "위클리 도구"가 아니라 패턴이다 — **흩어진 맥락 합성**: N개 소스 × M개 엔티티(다대다)에 흩어진 맥락을, 한 곳에서 신뢰가능한 단일 뷰로 합성하기.

**"내 문제가 이 패턴인가?"** — 맥락이 두 곳 이상에 흩어졌나, 엔티티가 다대다로 얽혔나, 단일 신뢰 뷰가 필요한가. 셋 다 그렇다면 위 네 결정이 그대로 적용된다:

- **브랜드 GTM 에이전트** — 한 브랜드를 시장에 올리는 데 필요한 맥락(채널·상품·고객·시장 신호)이 여러 소스에 흩어지고 상품×채널이 다대다. 흩어진 신호를 한 흐름으로 엮어 '전체가 흘러가게' 하는 게 핵심 (우리가 PoC로 돌리는 중인 사업부 1호 에이전트가 정확히 이 모양).
- **프로젝트·이슈 통합 현황** — 상태가 노션·git·슬랙·지라에 흩어짐.
- **사업부 머천다이징/영업 리포트** — 매장×상품×기간 다대다.
- **회의 결정 원장** — 결정이 여러 회의·채널·문서에 흩어짐.
- (AI 팀) **RAG·지식 합성**, **멀티에이전트 오케스트레이션**(나누고 합치고·중간 형식 = 같은 결정), **개인 브리핑**.

> 다대다·다소스 합성은 *모델을 키우는 문제가 아니라 구조를 정하는 문제*다. 위 네 결정을 당신 도메인에 맞춰 내리면 된다.

## 결론

흩어진 다대다 맥락을 엮는 일에서, 결과를 가른 건 모델이 아니라 구조였다. 일찍 나누되 압축은 마지막에, 수집은 접근 권한이 포괄하면 중앙으로, 단계 사이는 명시적 형식으로, 누락은 있어야 할 목록을 먼저 세워 막는다 — 이 네 결정이 품질과 비용을 갈랐다. 그 위에, 입력 자체의 구조가 좋아야 한다는 전제는 변함없다.

남은 과제도 분명하다. 경영 결정(이사님) 레이어 보강, 입력 품질의 개인별 편차를 줄일 일지 작성 표준화, 타인 git의 중앙 커버, 그리고 무거운 주에 자동화 시간이 스케줄러 한도에 닿는 문제. 다음 글감이다.
